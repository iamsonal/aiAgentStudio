# Standard Actions

Standard actions are the packaged tool implementations shipped in the core framework. Admins expose
them to the model by creating `AgentCapability__c` records with descriptions, backend config, and
parameter schemas.

## Capability Contract

Every capability built on a standard action still depends on the same core fields:

- `Description__c` tells the model when to use the tool
- `Parameters__c` defines the runtime input schema
- `BackendConfiguration__c` holds admin-owned operating settings
- `HITLMode__c` and `ExposureLevel__c` control review and visibility

## Core Packaged Standard Actions

  </Card>

  </Card>
  <Card title="HttpCallout" icon="seti:cloud">

  </Card>

<Aside type="note" title="Not the same as implementation type">
`Flow` and `Agent` are valid capability implementation types, but they are not packaged standard
action types in the same sense as the core actions listed here.
</Aside>

## Action Reference

| Action | What admins configure | What the model supplies |
| :-- | :-- | :-- |
| `GetRecordDetails` | object, field list or field set | one identifier field |
| `CreateRecord` | object and default field values | `recordData` object |
| `UpdateRecord` | optional object type and default values | `recordId` and `recordData` |
| `PostChatter` | feed type and target ID | `messageText` and optional topics |
| `SendEmail` | variant, sender defaults, template defaults | recipients plus direct-content or template arguments |
| `HttpCallout` | named credential, endpoint path, method, headers | path variables, query params, optional body |

## Design Guidance

### Descriptions matter more than most teams expect

The capability description is one of the strongest signals the model sees when it decides whether to
call a tool.

Good descriptions usually include:

- when to use the capability
- when not to use it
- what identifier or input the user should provide
- examples of correct usage

### Start narrow

- prefer one business purpose per capability
- keep field lists intentionally small
- use enums and tight schemas where possible
- add write actions only after retrieval behavior is stable

### Add review for side effects

For `CreateRecord`, `UpdateRecord`, `SendEmail`, and `HttpCallout`, decide up front whether
confirmation or approval is required in your workflow.

## Common Mistakes

- exposing too many write capabilities at once
- using vague descriptions such as “updates records”
- treating backend config as model-controlled input
- allowing external callouts without clear approval policy
- assuming `Flow` is a packaged standard action type
