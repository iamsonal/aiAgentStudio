# Architecture Overview

AI Agent Studio combines multiple agent patterns, workflow controls, trust layers, and observability into a single Salesforce-native framework. This page focuses on the capabilities users get from the framework rather than the internal implementation details.

<Aside type="note" title="Scope">
This page is intentionally feature-focused. It avoids internal class-level and object-level implementation detail while still describing the full functionality available in the framework experience.
</Aside>

<Aside type="note" title="Core vs Addons">
The open-source core runtime is centered on conversational agents. Function agents, email agents, sequential pipelines, and several specialized UIs are available only when addon metadata and orchestrator mappings are deployed.
</Aside>

## Agent Patterns

<CardGrid>
  <Card title="Conversational Agents" icon="chat">
    Multi-turn assistants for support, sales, service, and record-aware conversations. These agents combine memory, tools, approvals, and rich context to help users over time instead of in a single response.
  </Card>
  <Card title="Function Agents" icon="seti:gear">
    Focused agents for targeted jobs such as enrichment, classification, routing, scoring, and guided business actions where the user wants a fast, purpose-built result.
  </Card>
  <Card title="Email Agents" icon="seti:mail">
    Email-native workflows for triage, draft generation, routing, follow-up, and customer communication, all governed by the same trust and audit controls as other agent experiences.
  </Card>
  <Card title="Sequential Pipelines" icon="seti:graph-line">
    Fixed multi-agent workflows for predictable business processes where each stage has a clear responsibility and output.
  </Card>
  <Card title="Sub-Agents" icon="seti:git-branch">
    Specialized agents can be invoked from within a broader workflow when a task needs a different toolset, prompt strategy, or decision boundary.
  </Card>
</CardGrid>

## What The Framework Coordinates

<Steps>

1. **Request Intake**

   Requests can originate from chat, automation, APIs, email scenarios, or higher-level workflow compositions. The framework routes each request into the right execution pattern.

2. **Prompt and Context Preparation**

   The framework assembles identity, instructions, memory, business context, and available actions so the model has the information needed to respond or take action.

3. **Model Decision-Making**

   The model can reply directly, choose tools, request clarification, or move the process into a governed workflow that includes approvals or follow-up steps.

4. **Workflow Execution**

   The framework executes the required actions, coordinates multi-step progress, and continues until the work is complete or needs human intervention.

5. **Tracking and Completion**

   Every run is tracked in a durable execution history so teams can monitor status, inspect what happened, review reasoning, and understand cost and performance.

</Steps>

## Key Capabilities

<CardGrid>
  <Card title="Tool Use" icon="seti:tools">
    Agents can search data, read details, create records, update records, call flows, and run custom business logic.
  </Card>
  <Card title="Memory and Context" icon="seti:database">
    Agents can use prior conversation history, summarized memory, and related Salesforce context to stay grounded in the business workflow.
  </Card>
  <Card title="Approvals and Governance" icon="approve-check">
    Sensitive actions can require confirmation or approval before they execute, which helps teams safely automate higher-impact workflows.
  </Card>
  <Card title="Observability" icon="seti:graph-line">
    Execution histories, reasoning capture, and token usage data provide transparency for production operations. The storyboard-style UI is addon-provided.
  </Card>
  <Card title="Trust Layers" icon="seti:shield">
    PII masking, prompt safety, and tool-flow controls help keep agent behavior aligned with enterprise security and reliability expectations.
  </Card>
  <Card title="Provider Flexibility" icon="seti:cloud">
    The framework supports OpenAI-compatible APIs and broader enterprise provider strategies so teams can align AI choices with their platform requirements.
  </Card>
</CardGrid>

## Configuration Areas

<Tabs>
  <TabItem label="Agent Setup">

Configure the overall behavior of an agent:

<ul>
  <li>identity and instructions</li>
  <li>provider and model choice</li>
  <li>memory behavior</li>
  <li>trust controls</li>
  <li>async preferences</li>
  <li>reasoning and observability settings</li>
</ul>

  </TabItem>
  <TabItem label="Capabilities">

Define what an agent can do:

<ul>
  <li>which actions are available</li>
  <li>when each action should be used</li>
  <li>the inputs it expects</li>
  <li>whether it runs immediately or in the background</li>
  <li>whether it requires confirmation, approval, or conditional approval</li>
</ul>

  </TabItem>
  <TabItem label="Workflow Controls">

Control how the framework behaves operationally:

<ul>
  <li>approvals</li>
  <li>ordered tool flows</li>
  <li>execution limits</li>
  <li>routing and scale settings</li>
  <li>retry and resilience behavior</li>
</ul>

  </TabItem>
  <TabItem label="Observability">

Review and analyze what happened:

<ul>
  <li>execution status</li>
  <li>step history</li>
  <li>approval outcomes</li>
  <li>token and cost metrics</li>
  <li>reasoning and decision traces</li>
</ul>

  </TabItem>
</Tabs>

## Design Principles

<CardGrid>
  <Card title="Built for Salesforce" icon="warning">
    The framework is designed around Salesforce security, transaction rules, and governor limits so AI workflows remain practical in real orgs.
  </Card>
  <Card title="Safe Automation" icon="rocket">
    Multi-step workflows can continue across turns, tools, approvals, and background work while preserving control and auditability.
  </Card>
  <Card title="Deterministic Guardrails" icon="seti:history">
    Ordered tool flows, approval gates, and trust layers help keep agent behavior predictable in sensitive workflows.
  </Card>
  <Card title="Enterprise Visibility" icon="seti:git">
    Every run is observable so teams can troubleshoot, optimize prompts, monitor spend, and build confidence in production use.
  </Card>
</CardGrid>
