# Security & Governance

AI Agent Studio applies defense in depth across user context, data access, model interactions, human approvals, and auditability.

<Aside type="note" title="Scope">
This page focuses on user-facing security capabilities rather than internal implementation details.
</Aside>

## Security Architecture

<CardGrid>
  <Card title="Platform-Native Security" icon="seti:lock">
    Execution stays inside Salesforce security boundaries: user context, sharing, CRUD, field-level access, and approval processes.
  </Card>
  <Card title="Trust Layers" icon="seti:shield">

  </Card>

## Platform-Native Controls

### User context and sharing

- Agents execute in the initiating user context by default.
- Record visibility follows Salesforce sharing rules.
- When service-user routing is configured, the original user context is still preserved for audit and tracking.

<Aside type="tip" title="Service User Context">
Use service-user routing only when a workflow genuinely needs elevated execution context. Keep the service user narrowly permissioned.
</Aside>

### CRUD and FLS enforcement

- Queries should respect user access, commonly through `WITH USER_MODE`.
- DML should strip inaccessible fields before save and prefer `Database.*(..., AccessLevel.USER_MODE)` when the action is intended to run in user context.

```apex
List<Account> accounts = [
    SELECT Id, Name
    FROM Account
    WHERE Id IN :accountIds
    WITH USER_MODE
];
```

```apex
SObjectAccessDecision decision = Security.stripInaccessible(
    AccessType.UPDATABLE,
    recordsToSave
);
Database.update(decision.getRecords(), false, AccessLevel.USER_MODE);
```

## Trust Layers

<Tabs>
  <TabItem label="PII Masking">

PII masking reduces the chance that sensitive values leave Salesforce in raw form.

- Hybrid approach: schema-based detection plus pattern-based masking
- Session-scoped token mapping for deterministic mask and unmask behavior
- Applies to model-facing content and relevant tool arguments

Configuration:

- `PIIMaskingPreset__c = Off`: disabled
- `PIIMaskingPreset__c = Standard`: common categories and classifications
- `PIIMaskingPreset__c = Strict`: broader masking coverage

Example:

```text
Original: Update the case for SSN 123-45-6789
Masked:   Update the case for SSN [SSN:001]
```

  </TabItem>
  <TabItem label="Prompt Safety">

Prompt safety is delegated to the provider adapter so checks align with the provider's own moderation model.

<ul>
  <li>pre-checks can happen before the main provider request</li>
  <li>providers can also return safety signals as part of the response</li>
  <li>organizations can choose how broadly to enable these checks per agent</li>
</ul>

Configuration:

<ul>
  <li><code>PromptSafetyPreset__c = Off</code>: skip pre-check</li>
  <li><code>PromptSafetyPreset__c = Standard</code>: run native safety behavior when the adapter supports it</li>
</ul>

  </TabItem>
  <TabItem label="Tool Flow Graph">

The Tool Flow Graph constrains which tools the LLM can see at each step.

<ul>
  <li>entry tools are initially visible</li>
  <li>downstream tools unlock only after prerequisite tools succeed</li>
  <li>exclusive branches permanently lock siblings once a branch is chosen</li>
</ul>

This is configured on the agent through:

<ul>
  <li><code>EnableToolFlowGraph__c</code></li>
  <li><code>ToolFlowGraph__c</code></li>
</ul>

At runtime, the framework exposes only the tools that are currently eligible.

  </TabItem>
</Tabs>

## Human-in-the-Loop

Sensitive actions can require user confirmation or formal approval before they execute.

<CardGrid>
  <Card title="Confirmation" icon="chat">
    The agent asks the user to confirm before proceeding. This is conversational and does not require a formal approval record. It is intended for conversational agents.
  </Card>
  <Card title="Approval" icon="approve-check">
    The framework pauses execution and resumes only after an approver acts.
  </Card>
  <Card title="ConfirmationThenApproval" icon="warning">
    Combines conversational confirmation with formal approval for higher-risk actions. This mode is conversational-agent only and is incompatible with async tool execution.
  </Card>
  <Card title="ConditionalApproval" icon="seti:settings">
    Lets the workflow decide at runtime whether approval should be required based on the capability description and LLM decisioning.
  </Card>
</CardGrid>

Operationally:

- approvals pause the execution
- `HITLNotificationPreference__c` controls whether users are notified on approval and rejection outcomes for approval-based modes
- approvers must have access to the underlying business context
- outcomes are written back into the execution trace for auditability

## Audit and Observability

Every execution produces a durable trace.

Captured data can include:

- user messages
- assistant responses
- tool calls and tool results
- failures
- token counts and estimated cost
- tool rationale when reasoning capture is enabled

This gives teams a concrete review path for support, compliance, and prompt/tool tuning.

## Recommended Practices

<Steps>

1. **Start in sandbox**

   Validate CRUD, FLS, sharing, approvals, and provider behavior using realistic users and data.

2. **Use least privilege**

   Keep service users, integration users, and approvers scoped to the minimum permissions needed.

3. **Turn on trust layers incrementally**

   Start with masking and approval controls, then add prompt safety and tool-flow constraints after validating provider behavior.

4. **Protect high-impact actions**

   Use approvals for deletes, external side effects, financial updates, and irreversible workflows.

5. **Monitor execution data**

   Watch for unusual failure rates, token spikes, approval rejections, and prompt-safety events.

</Steps>

## Security Checklist

<Aside type="caution" title="Production Readiness">
Test with production-like data volume and user profiles before rollout. Make sure you can quickly disable or restrict agents if behavior is unexpected.
</Aside>
